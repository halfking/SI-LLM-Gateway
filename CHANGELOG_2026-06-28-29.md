## 🚀 二、可靠性修复（Reliability Fixes）

> 解决 2 个生产事故：上游挂起导致的 3-4 分钟/请求卡死 + 客户端错误的 503 误报。

### 2.1 P0 · 上游挂起硬超时（`9c614f44`）

**核心能力**：当上游（特别是通过企业代理 `172.31.0.2:7890` 转发）静默接受 TCP 连接但不响应时，**网关在 upCtx 到期时立即返回 503**，
不再被 `http.Client.Timeout` 失效问题拖住。

**关键改动**：
- `routing/executor_chat.go:988-1110` — 新增 `doUpstreamWithHardTimeout` + `doUpstreamRawWithHardTimeout`
- `routing/executor_chat.go:375, 377` — 替换原 `e.Upstream.Do(req)` / `httpClient.Do(req)` 调用点
- `routing/hard_timeout_test.go` — 新增 3 个单元测试

**设计要点**：goroutine + upCtx 桥接 + 5s watchdog 兜底；**绝不在 happy path 上 defer cancel callCtx**（否则会误报 `first_byte_timeout`）。

**性能与影响**：
- 修复前：qwen3-235b-a22b 等模型首次响应 200s+ 挂起；断路器打开需 360s+
- 修复后：首次 130s 返回 503，断路器 44ms 返回
- 兼容性：100% 向后兼容

### 2.2 P1 · missing_model 错误码 503→400（`a1417474`）

**核心能力**：`POST /v1/chat/completions` 当 `model` 字段缺失/空/空白/null 时，**统一返回 400 missing_model**（与 messages / responses handler 一致）。

**关键改动**：
- `relay/handler.go:453-490` — 新增 peek-and-replace 校验块
- `relay/handler.go:854-862, 2212, 2258` — 配套日志/错误码
- `relay/handler_missing_model_test.go` — 4 个子用例测试

**修复效果**（生产验证）：

| 输入 | 修复前 | 修复后 |
|------|--------|--------|
| 字段缺失 | 503 no_candidate | **400** 0.10s |
| 空串 | 503 no_candidate | **400** 0.07s |
| 空白 | 503 no_candidate | **400** 0.09s |
| null | 503 no_candidate | **400** 0.10s |

**兼容性**：行为变更（错误码 503→400），语义上更正确；依赖原 503 走重试的客户端需适配。

## 🔌 三、会话与重连（Sessions & Reconnect）

> 修"空响应"事故 + 构建断线重连基础设施 + 网络重试自动去重。
> 三个 commit 共同形成一个完整的"客户端掉线不丢消息"方案。

### 3.1 会话统一管理（`7576c021` 的 fix 部分）

**核心能力**：3 个端点（chat / messages / responses）的会话管理逻辑**统一**到 `resolveSessionFromRequest()`；会话过期实现**真正的滑动窗口**（Touch 同时刷新 `expires_at` + Redis TTL）。

**关键改动**：
- `relay/session_resolution.go`（249 行，新文件）— 统一会话解析 + fallback 创建
- `sessions/session.go`（修改 +27 行）— `Touch()` 滑动续期
- `relay/messages.go` / `relay/responses.go`（修改 +42 / +37 行）— 接入会话上下文
- `relay/request_context.go`（修改 +15 行）— `gwSessionTaskFromRequest()` 识别所有 session 头（X-Conversation-Id / X-Chat-Session-Id / X-Thread-Id）
- `relay/session_resolution_test.go` + `sessions/session_test.go` — 共 17 个测试

**业务价值**：修"模型未返回任何内容"但 `request_logs` 缺记录的事故。

### 3.2 断线重连配置框架（`7576c021` 的 feat 部分）

**核心能力**：新增 `reconnect/` 包 + admin API，支持**全局 + 租户级**配置；默认 `Enabled=false`，按需开启。

**关键改动**：
- `reconnect/config.go`（167 行，新文件）— `Config` / `TenantConfig` / `Manager` 三个核心类型
- `reconnect/config_test.go`（152 行，8 个测试）
- `admin/reconnect_config.go`（87 行，新文件）— `GET/POST /api/reconnect/config` 端点

**关键配置**：

| 字段 | 默认 | 含义 |
|------|------|------|
| `Enabled` | `false` | 总开关 |
| `AutoResumeByDefault` | `false` | 是否自动 resume |
| `CacheTTL` | `168h` (7d) | 响应缓存 TTL |
| `MaxCacheBodyBytes` | `1048576` (1 MiB) | 响应 body 上限 |

### 3.3 基于内容的请求去重（`b6ea9ff6`）

**核心能力**：客户端网络闪断自动重试时，**即使生成了新 request ID**，网关通过 SHA-256 指纹（last 3 messages + model + stream）识别出**内容相同**的请求，**直接重放缓存响应**。

**关键改动**：
- `relay/content_dedup.go`（214 行，新文件）— `ContentDedupCache` + 9 个公开方法
- `relay/content_dedup_test.go`（193 行，11 个测试）
- `relay/handler.go:158-163, 235-240, 1325-1370`（+54 行）— `ChatHandler` 集成
- `reconnect/config.go:48+`（+33 行）— 追加 `ContentDedup*` 配置字段
- `CONTENT_DEDUP_ANALYSIS.md`（311 行）+ `CONTENT_DEDUP_IMPLEMENTATION.md`（444 行）

**关键配置**（追加到 `reconnect.Config`）：

| 字段 | 默认 | 含义 |
|------|------|------|
| `ContentDedupEnabled` | `false` | 是否开启去重 |
| `ContentDedupWindow` | `10m` | 去重时间窗口 |
| `ContentDedupDepth` | `3` | 参与指纹的消息数（last N） |

**性能**：
- 指纹计算：~1-5μs
- 缓存检查开销：< 3ms
- 命中响应：瞬时（vs 2-10s LLM 调用）
- 预期命中率：60-80%（网络重试场景）

**⚠️ 依赖关系**：`b6ea9ff6` 在 `reconnect/config.go` 上**追加**字段，**必须**先 cherry-pick `7576c021`。
