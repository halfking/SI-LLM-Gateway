# F3 审计 — app.current_tenant GUC (RLS Defense-in-Depth)

> **Date**: 2026-06-22
> **Commit**: f6214397 (fix(rls): F3 withTenantTx)
> **Status**: ⚠️ Partially effective (DB user bypass)

## 问题

PostgreSQL RLS policies on `request_logs`, `tenant_tool_policies`,
`tenant_model_policies` etc. use `get_current_tenant()` which reads
the `app.current_tenant` GUC. The GUC was never set by Go code,
so RLS always returned 'default' — accidentally blocking non-default
tenants from accessing their own data.

## 修复

1. `admin/tenant_ctx.go` — new helper `withTenantTx(ctx, pool, tenantID, fn)`
   that wraps a callback in a read-only transaction with
   `SET LOCAL app.current_tenant = '<escaped_tenant_id>'`.
2. Applied to 6 query locations in `session_list.go` and `session_compare.go`.
3. Single quotes escaped to prevent SQL injection.

## ⚠️ 关键发现 (2026-06-22 部署后验证)

`llm_gateway` DB user is **SUPERUSER** (`rolsuper=t`):

```sql
SELECT rolname, rolsuper, rolbypassrls FROM pg_roles WHERE rolname='llm_gateway';
  rolname   | rolsuper | rolbypassrls
-------------+----------+--------------
 llm_gateway | t        | t
```

**PostgreSQL superusers ALWAYS bypass RLS**, regardless of:
- `bypassrls` attribute (revoked 2026-06-22)
- `FORCE ROW LEVEL SECURITY` (applied 2026-06-22 to 8 tables)

**Result**: `withTenantTx()` sets the GUC correctly, but RLS policies
do NOT enforce because the user is a superuser.

## 缓解

1. **Application-level filter still works**: All queries use
   `WHERE tenant_id = $1` with `EffectiveTenantID(r)`.
2. **withTenantTx() is still correct code**: It would work if the
   user were downgraded from superuser.
3. **FORCE ROW LEVEL SECURITY applied**: If the user is ever
   downgraded, RLS will immediately become effective.

## 建议的后续步骤

1. **生产环境**: 评估是否可以将 `llm_gateway` 从 superuser 降级。
   - 风险: 迁移脚本和某些 DDL 操作可能需要 superuser
   - 缓解: 创建一个独立的 `llm_gateway_admin` 用于迁移
2. **替代方案**: 在应用层做更严格的租户隔离审计（已完成 F2）
3. **监控**: 跟踪 `withTenantTx` 的执行日志，确保 GUC 被设置

## ✅ 最终修复 (2026-06-22)

### 角色分离方案

| 角色 | 权限 | 用途 |
|------|------|------|
| `llm_gateway` | NOSUPERUSER, NOBYPASSRLS | 应用运行时 (pool) |
| `llm_gateway_admin` | SUPERUSER | 数据库迁移 |

### 验证结果

```sql
-- 角色状态
SELECT rolname, rolsuper, rolbypassrls FROM pg_roles WHERE rolname LIKE 'llm_gateway%';
      rolname      | rolsuper | rolbypassrls
-------------------+----------+--------------
 llm_gateway_admin | t        | f
 llm_gateway       | f        | f

-- RLS 现在生效！
BEGIN; SET LOCAL app.current_tenant = 'hansi';
SELECT count(*) FROM request_logs WHERE ts > NOW() - INTERVAL '1 day';
-- 返回 15 (hansi 的真实数据)

BEGIN; SET LOCAL app.current_tenant = 'acme';
SELECT count(*) FROM request_logs WHERE ts > NOW() - INTERVAL '1 day';
-- 返回 0 (acme 不存在，RLS 阻止访问)

-- 应用正常启动
healthz=200
```

### 后续

- 迁移脚本如需超级用户权限，使用 `llm_gateway_admin` 连接
- 应用运行时使用 `llm_gateway` (非超级用户，RLS 生效)
- 如需回滚: `ALTER USER llm_gateway WITH SUPERUSER;`
