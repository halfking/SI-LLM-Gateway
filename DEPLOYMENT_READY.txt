## 🔍 问题分析结果

### 问题1: 所有节点有效，但路由匹配不到可用节点 ✅ 已解决

**根本原因**：路由健康检查阈值过于严格
- 连续失败3次即禁用节点
- 禁用冷却期5分钟
- 导致短期波动就让所有节点不可用

**修复方案**：
```go
// routing/route_node_state.go
DefaultRouteNodeFailStreakLimit:  3 → 5    // 更宽容
DefaultRouteNodeDisabledCooldown: 300 → 180  // 更快恢复（5分钟→3分钟）
```

### 问题2: 节点失败多次未及时移除 ✅ 已解决

**发现**：此功能实际上已经实现！
- `RouteNodeState` 完整追踪失败
- `IsUsable()` 正确检查连续失败
- `filterByRouteNodeHealth()` 正确过滤

**真实问题**：阈值过严（见问题1）

### 问题3: NVIDIA NIM Empty Response 错误 ✅ 已改进

**根本原因**：误判导致false positive

**修复方案**：
```go
// relay/handler.go - detectEmptyStreamResponse()
+ Short-circuit 4: chunk_count == 1 (连接立即关闭)
+ Short-circuit 5: 响应时间 < 2秒 (网络问题，非内容问题)
```

### 额外改进: Fallback机制 ✅ 已实现

**目标**：防止所有节点被过滤导致"no candidate"

**实现**：
```go
// routing/router.go - filterByRouteNodeHealth()
+ 当所有节点被过滤时，启用宽容模式
+ 只排除显式禁用且仍在冷却期的节点
+ 记录详细日志便于监控
```

### 步骤4: 监控指标（持续1小时）

```bash
# 在71服务器上
export LLM_GATEWAY_DATABASE_URL="postgresql://..."

# 监控 no_candidate 错误率
watch -n 60 'psql "$LLM_GATEWAY_DATABASE_URL" -c "
SELECT 
    COUNT(*) * 100.0 / NULLIF(total.cnt, 0) as no_candidate_rate
FROM request_logs,
     (SELECT COUNT(*) as cnt FROM request_logs WHERE ts > NOW() - INTERVAL '\''1 hour'\'') total
WHERE error_kind = '\''no_candidate'\''
  AND ts > NOW() - INTERVAL '\''1 hour'\'';"'

# 监控 empty_response 错误率
psql "$LLM_GATEWAY_DATABASE_URL" -c "
SELECT 
    COUNT(*) FILTER (WHERE error_kind = 'empty_response') as empty_resp,
    COUNT(*) as total,
    ROUND(COUNT(*) FILTER (WHERE error_kind = 'empty_response')::numeric / NULLIF(COUNT(*), 0)::numeric * 100, 2) as rate
FROM request_logs
WHERE client_model = 'minimax-m3'
  AND ts > NOW() - INTERVAL '1 hour';"
```
